**“基于网络协议的工况模拟与数据交互”**。

既然是**“基于MQTT的车载环境传感器测试系统”**，这里的“环境传感器”在汽车行业通常指代<u>**车内空气质量系统（IAQS）相关的传感器**，比如：车内温度、车外温度、湿度、PM2.5、CO2浓度等</u>。

这个模拟传感器变成一个**独立的边缘节点（Python脚本）**，它既能自己跑工况，又能随时接收你MQTT测试指令的改变。

---

### 一、 整体架构定位

在你的毕设系统中，这个模块处于最底层的**感知层**：
1.  **模拟传感器节点 (Python脚本)**：运行在树莓派、本机或Docker中，**<u>负责生成带有物理意义的数据</u>**。
2.  **MQTT Broker (如EMQX/Mosquitto)**：中间件，负责数据路由。
3.  **你的测试系统上位机 (Web/Python客户端)**：订阅数据做可视化，发送指令控制传感器。

### 二、 核心设计：MQTT主题与JSON数据流

汽车电子越来越倾向于使用JSON（比如部分车联网V2X协议和SOME/IP的衍生），对于毕设来说，JSON的可读性极强，非常方便你写论文和答辩。

#### 1. 上行数据主题（传感器 -> 测试系统）
*   **Topic:** `vcar/sensors/environment/data`
*   **Payload (JSON):**
```json
{
  "timestamp": 1698765432100,
  "sensor_id": "ENV_SENSOR_001",
  "data": {
    "in_car_temp": 26.5,       // 车内温度 (℃)
    "out_car_temp": 35.2,      // 车外温度 (℃)
    "humidity": 45.0,          // 车内湿度 (%RH)
    "pm25": 15.2,              // PM2.5 (ug/m³)
    "co2": 450.0               // CO2浓度
  },
  "status": "NORMAL",          // 系统状态：NORMAL / FAULT / WARN
  "fault_code": "NONE"         // 故障码：NONE / SENSOR_STUCK / OPEN_CIRCUIT
}
```

#### 2. 下行控制主题（测试系统 -> 模拟传感器）
这是你用来“人为控制工况”的接口。
*   **Topic:** `vcar/sensors/environment/control`
*   **Payload (JSON):**
```json
// 场景1：切换预设的车载工况
{
  "command": "set_scenario",
  "params": {
    "scenario": "summer_traffic_jam"  // 夏日拥堵工况（温度升高、CO2飙升、PM2.5升高）
  }
}

// 场景2：注入特定故障
{
  "command": "inject_fault",
  "params": {
    "target": "pm25",
    "fault_type": "STUCK",            // PM2.5传感器卡滞
    "stuck_value": 75.0               // 卡在75
  }
}

// 场景3：手动微调某个值（用于边界测试）
{
  "command": "override_value",
  "params": {
    "target": "in_car_temp",
    "value": -45.0                    // 模拟极寒超下限异常值
  }
}
```

### 三、 工况场景库设计（毕设亮点）

不要只提供随机数！在论文里，**“工况设计”是体现你汽车专业素养的核心**。你可以在代码里预置几个典型的汽车行驶工况：

1.  **`static_parking_summer` (夏日静置)**：车内温度从35℃慢慢升至65℃，车外35℃，CO2=400。
2.  **`winter_cruising` (冬季高速巡航)**：车内稳定在22℃，车外-10℃，湿度极低(15%)，PM2.5受过滤影响为5。
3.  **`tunnel_following` (隧道跟车)**：车外温度正常，但PM2.5突然飙升至150+，CO2逐渐升至1500+（模拟前车尾气和空调外循环）。
4.  **`highway_ac_leak` (高速空调漏气)**：车内温度缓慢上升，湿度异常飙升到80%以上。

### 四、 核心代码实现 (Python + paho-mqtt)

这是一个完整的、可以直接跑的毕设感知层节点代码骨架，集成了物理逻辑、MQTT通信和工况切换：

```python
import paho.mqtt.client as mqtt
import json
import time
import random
import math
import threading

class CarEnvironmentSimulator:
    def __init__(self, broker_ip, port=1883):
        self.broker_ip = broker_ip
        self.port = port
        
        # 传感器真实物理状态 (带惯性)
        self.state = {
            "in_car_temp": 25.0,
            "out_car_temp": 25.0,
            "humidity": 50.0,
            "pm25": 20.0,
            "co2": 400.0
        }
        
        # 目标值 (工况切换时改变目标，物理状态会慢慢逼近目标)
        self.targets = self.state.copy()
        
        # 故障状态
        self.status = "NORMAL"
        self.fault_code = "NONE"
        self.overrides = {} # 手动强制覆盖的值
        
        # MQTT客户端
        self.client = mqtt.Client(client_id="EnvSimulator_001")
        self.client.on_connect = self.on_connect
        self.client.on_message = self.on_message

    def on_connect(self, client, userdata, flags, rc):
        print(f"连接到MQTT Broker成功, 状态码: {rc}")
        # 连接成功后，订阅控制主题
        client.subscribe("vcar/sensors/environment/control")
        print("已订阅控制主题: vcar/sensors/environment/control")

    def on_message(self, client, userdata, msg):
        try:
            payload = json.loads(msg.payload.decode())
            print(f"收到控制指令: {payload['command']}")
            
            if payload["command"] == "set_scenario":
                self.apply_scenario(payload["params"]["scenario"])
            elif payload["command"] == "inject_fault":
                self.inject_fault(payload["params"])
            elif payload["command"] == "override_value":
                self.overrides[payload["params"]["target"]] = payload["params"]["value"]
            elif payload["command"] == "clear_fault":
                self.clear_fault()
                
        except Exception as e:
            print(f"解析控制指令失败: {e}")

    def apply_scenario(self, scenario_name):
        """切换工况目标值"""
        self.clear_fault()
        if scenario_name == "summer_traffic_jam":
            self.targets = {"in_car_temp": 38.0, "out_car_temp": 35.0, "humidity": 60.0, "pm25": 120.0, "co2": 1800.0}
        elif scenario_name == "winter_cruising":
            self.targets = {"in_car_temp": 22.0, "out_car_temp": -10.0, "humidity": 15.0, "pm25": 5.0, "co2": 450.0}
        elif scenario_name == "tunnel_following":
            self.targets = {"in_car_temp": 24.0, "out_car_temp": 26.0, "humidity": 45.0, "pm25": 180.0, "co2": 2200.0}
        else:
            print(f"未知工况: {scenario_name}")

    def inject_fault(self, params):
        """注入故障"""
        self.status = "FAULT"
        target = params.get("target")
        fault_type = params.get("fault_type")
        
        if fault_type == "STUCK":
            self.fault_code = f"{target.upper()}_STUCK"
            self.overrides[target] = params.get("stuck_value", self.state[target])
        elif fault_type == "OPEN_CIRCUIT":
            self.fault_code = f"{target.upper()}_OPEN_CIRCUIT"
            # 断路通常表现为0或极小值
            self.overrides[target] = 0.0 
        elif fault_type == "SHORT_CIRCUIT":
            self.fault_code = f"{target.upper()}_SHORT"
            # 短路通常表现为最大量程
            max_vals = {"pm25": 999, "co2": 5000, "in_car_temp": 80, "humidity": 100}
            self.overrides[target] = max_vals.get(target, 999.0)

    def clear_fault(self):
        self.status = "NORMAL"
        self.fault_code = "NONE"
        self.overrides = {}

    def update_physics(self):
        """模拟物理惯性 (一阶低通滤波)"""
        tau = 2.0 # 惯性时间常数
        dt = 1.0  # 假设1秒更新一次
        alpha = dt / (tau + dt)
        
        for key in self.state:
            if key not in self.overrides: # 没有被强制覆盖或故障注入的才更新物理
                noise = random.gauss(0, 0.05 * self.targets[key]) # 加一点白噪声
                self.state[key] += alpha * (self.targets[key] - self.state[key]) + noise

    def publish_data(self):
        """组装JSON并发布"""
        final_data = self.state.copy()
        # 如果有强制覆盖的值（故障或手动），使用覆盖值
        final_data.update(self.overrides)
        
        payload = {
            "timestamp": int(time.time() * 1000),
            "sensor_id": "ENV_SENSOR_001",
            "data": final_data,
            "status": self.status,
            "fault_code": self.fault_code
        }
        
        self.client.publish("vcar/sensors/environment/data", json.dumps(payload), qos=1)
        print(f"发布数据: {payload['data']} | 状态: {self.status}")

    def run(self):
        self.client.connect(self.broker_ip, self.port, 60)
        self.client.loop_start()
        
        try:
            while True:
                self.update_physics()
                self.publish_data()
                time.sleep(1) # 1秒采样一次，适合环境类传感器
        except KeyboardInterrupt:
            self.client.loop_stop()
            print("模拟器停止")

if __name__ == "__main__":
    # 替换为你实际的MQTT Broker IP
    simulator = CarEnvironmentSimulator(broker_ip="127.0.0.1", port=1883)
    simulator.run()
```

### 五、 为什么这个设计在毕设答辩中能拿高分？

1.  **闭环完整性**：你的测试系统不仅能“看”（订阅数据），还能“控”（发布指令改变工况）。这叫**主动测试**，比单纯的被动监听高出一个档次。
2.  **物理真实性**：你在论文里可以写：“本系统采用一阶惯性环节模拟传感器的热力学/空气动力学滞后效应，避免了直接输出随机数导致的数据跳变，更贴近真实车况”。（导师最喜欢看这种有理论支撑的话）。
3.  **测试场景覆盖度**：你可以用这个模块在论文里画出三张图：
    *   **正常工况图**：下发 `winter_cruising` 指令后，温度曲线平滑下降的过程。
    *   **故障注入图**：下发 `inject_fault (OPEN_CIRCUIT)` 指令后，CO2瞬间掉到0，同时状态字段变红的截图。
    *   **极限边界图**：下发 `override_value` 让温度强行超出合理范围（比如-50度），测试你的上位机软件是否能正确触发“超下限报警”机制（这体现了你的测试系统具有**边界值分析**能力）。
