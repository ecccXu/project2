# 基于MQTT协议的车载环境传感器测试系统 - 需求梳理文档

# 一、需求核心目标

**基于MQTT的车载环境传感器测试系统**旨在构建一套**完整的自动化测试体系**，通过此系统实现传感器数据采集、MQTT传输、自动化测试引擎校验、数据存储及前端展示的闭环流程通过MQTT通信协议，实现对ESP32+DHT11采集数据的可靠传输，集成自动化测试引擎，能够对传感器数据格式、车载工作范围、MQTT通信可靠性等方面进行多维度校验并生成测试报告。

系统测试车载传感器，通过 MQTT 协议上报数据时，在各种情况下（正常、异常、高负载）的表现是否**符合车规要求**。

# 二、核心模块（必选功能，优先级：高）

核心模块以构建**完整的自动化测试体系**为核心，需实现“传感器数据采集-传输-测试系统处理-存储-前端展示”的完整闭环，同时严格遵循前后端分离原则（前端负责交互与展示，后端负责业务逻辑与数据处理）。

## 2.1 传感器数据采集模块

**功能描述**：分阶段实现温湿度数据采集，第一阶段（核心模块开发期）通过Python脚本模拟生成温湿度数据，支持模拟参数配置；第二阶段（核心模块完成后）预留真实传感器接入接口，实现ESP32+DHT11真实温湿度传感器的数据采集；所有数据统一封装为标准格式后通过MQTT协议发布至指定主题，支持数据采集频率配置。

**优先级**：高（核心数据来源，无此模块则测试无对象）

**前端需求**：

- 提供模式切换界面：支持“Python模拟模式”“真实传感器模式”（真实模式默认置灰，提示核心模块完成后启用）；

- 模拟模式配置项：支持采集频率（1-5秒）、模拟温湿度波动范围（温度：-40℃~85℃、湿度：0%~100%）、MQTT QoS级别（0/1/2）配置；

- 显示当前运行模式、传感器/模拟数据连接状态（已连接/未连接）、数据采集状态及实时温湿度数据；

- 显示数据发布日志（发布时间、主题、数据内容、数据来源：模拟/真实）。

**后端需求**：

- 第一阶段（核心模块开发期）：开发Python模拟脚本，生成符合车载场景的温湿度模拟数据，支持配置波动范围、采集频率，模拟数据异常值（用于测试系统校验）；

- 集成MQTT客户端（Eclipse Paho），实现与EMQ X Broker的连接、断线重连及数据发布，确保模拟数据与后续真实数据的发布逻辑一致；

- 将模拟/真实温湿度数据统一封装为标准JSON格式（包含sensorId、sensorType、temperature、humidity、timestamp、dataSource等字段）；

- 提供标准化接口，接收前端传入的模式切换指令、配置参数（模拟模式：采集频率、波动范围；真实模式：传感器采样率；通用：QoS级别）并执行；

- 实现数据采集状态、模式运行状态的后端监控，供前端查询；预留ESP32+DHT11真实传感器对接接口，核心模块完成后可直接迭代开发；

- 第二阶段（核心模块完成后）：实现ESP32与DHT11传感器的硬件对接，编写传感器驱动程序，完成真实数据采集与标准化转换，复用已有的MQTT发布及接口逻辑。

- 集成MQTT客户端（Eclipse Paho），实现与EMQ X Broker的连接、断线重连；

- 将采集的温湿度数据封装为标准JSON格式（包含sensorId、sensorType、temperature、humidity、timestamp等字段）；

- 根据前端配置参数，周期性将真实温湿度数据发布至指定MQTT主题（vehicle/sensor/data/{sensorId}）。

**接口需求**：

- MQTT连接接口（后端提供）：与EMQ X Broker的TCP连接接口（默认端口1883），供后端采集模块（Python模拟脚本/真实传感器采集程序）统一调用；

- 配置参数接口（前后端交互）：HTTP RESTful接口，前端传入模式类型、采集频率、模拟波动范围（模拟模式）、QoS级别等参数，后端接收并响应处理结果；

- 状态查询接口（前后端交互）：HTTP/WebSocket接口，前端查询当前运行模式、传感器/模拟数据连接状态、数据采集状态，后端实时返回。

## 2.2 MQTT通信核心模块

**功能描述**：实现MQTT协议的核心通信能力，包括消息的接收、路由、转发，确保传感器数据从发布端到测试系统的可靠传输。

**优先级**：高（系统核心通信链路，串联感知层与应用层）

**前端需求**：

- 显示MQTT Broker运行状态（已启动/未启动）；

- 展示当前活跃的MQTT客户端（发布端/订阅端）列表；

- 支持查看核心MQTT主题的消息流转日志。

**后端需求**：

- 部署EMQ X Broker（本地运行），支持MQTT 3.1.1/5.0协议；

- 实现消息路由功能：将发布端的传感器数据转发至订阅端（测试系统）；

- 支持消息持久化（针对QoS 1/2级别），确保断线后消息不丢失；

- 管理客户端会话，支持断线重连后的会话续传。

**接口需求**：

- Broker客户端连接接口（后端内部调用）：支持发布端、订阅端的CONNECT/CONNACK报文交互；

- 消息发布/订阅接口（后端内部调用）：支持PUBLISH/SUBSCRIBE/SUBACK/PUBACK等核心报文交互；

- 消息状态查询接口（前后端交互）：WebSocket接口，前端查询消息流转日志，后端实时推送。

## 2.3 自动化测试核心模块

**功能描述**：作为系统核心模块，负责接收MQTT传输的温湿度数据，加载并执行自动化测试用例，完成数据格式、范围、协议可靠性等校验，生成结构化测试结果，提供测试报告生成能力，同时为前端提供标准化接口支撑交互。

**优先级**：高（测试系统的核心能力，直接实现项目测试目标）

**前端需求**：

- 展示测试用例列表（用例ID、用例名称、测试类型、状态）；

- 支持测试用例的“批量执行”“单条执行”“暂停”操作；

- 实时展示测试结果（通过/失败）及失败原因（如“温度超出范围：90℃＞85℃”）；

- 支持测试报告的生成、预览及导出（Excel/HTML格式）。

**后端需求**：

- 集成MQTT订阅端，接收传感器数据并解析为标准格式；

- 基于Pytest框架构建测试用例库（覆盖数据格式校验、车载温湿度范围校验（温度-40℃~85℃、湿度0%~100%）、MQTT QoS可靠性校验）；

- 实现测试用例的自动化执行逻辑，支持用例优先级排序、批量/单条执行控制；

- 生成结构化测试结果（包含用例ID、sensorId、测试时间、结果、失败原因）；

- 支持测试报告的自动生成（包含测试覆盖率、通过率、详细结果列表）；

- 提供标准化接口，支撑前端测试用例管理、测试执行控制、结果查询及报告导出。

**接口需求**：

- 测试用例管理接口：支持用例的增删改查（供前端调用）；

- 测试执行接口：接收前端的测试操作指令（执行/暂停/停止）；

- 报告导出接口：提供Excel/HTML格式报告的下载接口。

## 2.4 数据存储模块

**功能描述**：为软件测试系统提供数据持久化支撑，负责存储传感器原始数据、测试用例、测试结果，提供高效的数据读写接口，供后端测试引擎调用及前端查询。

**优先级**：高（无持久化存储则无法追溯历史数据，测试报告无法生成）

**前端需求**：

- 支持按sensorId、时间范围查询历史传感器数据；

- 支持按测试时间、用例ID查询历史测试结果；

- 显示数据存储占用空间及存储状态。

**后端需求**：

- 基于SQLite构建数据库，设计核心表结构（sensor_data：传感器原始数据表；test_case：测试用例表；test_result：测试结果表）；

- 实现数据的增删改查操作，确保数据写入/查询的高效性；

- 支持数据按时间分区存储（可选，优化历史数据查询性能）；

- 实现数据备份功能（防止数据丢失）。

**接口需求**：

- 数据写入接口（后端内部调用）：接收测试引擎传入的传感器原始数据、测试结果，写入数据库；

- 数据查询接口（前后端交互）：HTTP RESTful接口，支持前端按sensorId、时间范围、用例ID等多条件查询数据，后端返回结构化结果；

- 数据备份接口（前后端交互）：HTTP接口，接收前端备份指令，后端执行备份并返回结果。

## 2.5 前端可视化模块（前后端分离架构-前端层）

**功能描述**：基于前后端分离架构，作为用户交互入口，通过调用后端标准化接口，实现传感器配置、测试系统操作、数据展示等功能，不参与业务逻辑处理。

**优先级**：高（用户交互入口，无可视化则无法便捷操作系统）

**前端需求**：

- 首页/仪表盘：展示系统整体状态（Broker状态、连接客户端数、当前测试进度、通过率）；

- 传感器配置页：对应“传感器数据采集模块”的前端交互界面，提供参数配置、状态查看功能；

- 测试管理页：对应“自动化测试引擎模块”的前端操作界面；

- 数据查询页：对应“数据存储模块”的前端查询界面；

- 系统设置页：Broker地址配置、数据存储路径配置、备份策略配置。

**后端支撑需求（前后端分离架构-后端服务层）**：

- 提供RESTful API和WebSocket服务，支撑前端所有交互操作（配置提交、测试执行、数据查询等）；

- 提供前端页面的静态资源服务（或通过Nginx部署前端资源，实现彻底分离）；

- 保存用户配置参数（如Broker地址、数据存储路径）至本地配置文件，供后端各模块调用。

**接口需求**：

- 页面资源接口：提供前端页面的HTML/CSS/JS资源；

- 实时数据推送接口：通过WebSocket推送系统状态、测试结果等实时数据；

- 用户配置接口：接收并保存用户的系统配置参数。

# 三、可选模块（迭代功能，均为软件测试系统的扩展能力）

可选模块用于提升系统含金量，适配考研复试（系统扩展性）、求职（车联网核心技能）的高阶需求，可根据开发时间灵活迭代。

## 3.1 性能测试模块

**功能描述**：作为软件测试系统的扩展能力，测试多温湿度传感器并发场景下的MQTT传输性能、测试系统处理能力，输出性能指标（消息延迟、吞吐量、丢包率），强化测试系统的全面性。

**优先级**：中（提升项目技术深度，匹配车企车联网性能测试岗位需求）

**前端需求**：

- 实时展示性能测试指标（消息延迟：平均/最大/最小；吞吐量：消息/秒；丢包率）；

- 支持性能测试结果的图表展示（折线图/柱状图）及导出。

**后端需求**：

- 添加性能指标统计逻辑（计算消息从发布到接收的延迟、单位时间内处理消息量、丢包数量）；

- 支持性能测试的压力场景模拟（如持续10分钟高并发数据传输）；

- 生成性能测试报告（包含测试场景、指标数据、分析结论）。

**接口需求**：

- 并发配置接口（前后端交互）：接收前端传入的并发节点数量、测试时长等参数；

- 性能指标查询接口（前后端交互）：WebSocket接口，实时返回性能测试过程中的指标数据；

- 性能报告导出接口（前后端交互）：HTTP接口，提供图表+文字的综合报告下载。

## 3.2 云端集成模块

**功能描述**：扩展软件测试系统的部署场景，将本地MQTT Broker替换为云端IoT平台（如阿里云IoT、华为云IoT），使测试系统具备远程传感器数据测试、多端访问能力。

**优先级**：中（展示云端物联网架构设计能力，适配车联网远程监控场景）

**前端需求**：

- 提供云端IoT平台配置界面（平台类型、接入地址、设备证书、密钥）；

- 支持“本地模式”与“云端模式”的切换；

- 展示云端连接状态、设备在线状态。

**后端需求**：

- 适配云端IoT平台的MQTT接入协议（支持TLS加密传输）；

- 实现设备身份认证逻辑（基于设备证书/密钥）；

- 支持云端主题的动态订阅与发布；

- 确保云端与本地数据的同步（可选）。

**接口需求**：

- 云端IoT平台接入接口：支持与阿里云IoT/华为云IoT的MQTT连接（端口8883，TLS加密）；

- 设备认证接口：与云端平台的身份校验接口。

## 3.3 真实传感器对接模块

**功能描述**：扩展软件测试系统的测试范围，支持接入其他类型车载环境传感器（如PM2.5传感器、CO₂传感器），使测试系统能适配多类型传感器的测试需求。

**优先级**：中（提升项目真实性，展示硬件对接能力，适配车企真实测试场景）

**前端需求**：

- 支持新增传感器类型选择（PM2.5/CO₂等）与参数配置（采样率、量程）；

- 显示新增真实传感器的连接状态、采样状态及实时数据；

- 测试用例列表支持新增传感器对应的测试项展示与操作。

**后端需求**：

- 实现ESP32与真实传感器的硬件接口对接（如I2C、UART）；

- 编写真实传感器的数据采集驱动程序；

- 兼容真实传感器的数据格式，转换为系统标准JSON格式；

- 支持模拟数据与真实数据的切换采集。

**接口需求**：

- 硬件接口：ESP32与真实传感器的I2C/UART通信接口；

- 数据转换接口：将传感器原始数据转换为系统标准格式的接口。

## 3.4 AI异常检测模块

**功能描述**：提升软件测试系统的智能性，基于历史温湿度数据和测试结果，通过机器学习模型预测温湿度传感器故障、识别异常数据模式，为测试系统增加智能诊断能力。

**优先级**：低（技术难度高，属于高阶扩展功能，适合有余力时迭代）

**前端需求**：

- 实时展示新增传感器采集的数据，与温湿度数据统一展示。

- 展示AI模型训练状态、准确率；

- 实时展示异常检测结果（如“温度传感器数据突变，疑似故障”）；

- 支持异常日志的查看与导出。

**后端需求**：

- 基于Python机器学习库（Scikit-learn/TensorFlow）构建异常检测模型；

- 实现模型的实时推理，对新增传感器数据进行异常判断；

- 支持模型的更新与重新训练。

**接口需求**：

- 模型训练接口（前后端交互）：接收前端传入的训练数据范围指令，后端触发模型训练并返回状态；

- 异常检测接口（前后端交互）：接收前端传入的实时温湿度数据，后端返回检测结果；

- 异常日志查询接口（前后端交互）：支持前端查询历史异常检测记录。

- 模型训练接口：接收训练数据，触发模型训练；

# 四、需求优先级说明（聚焦软件测试系统）

- 高优先级：核心模块（5个），以软件测试系统为核心，确保实现“数据采集-传输-测试-存储-展示”闭环，同时严格落实前后端分离架构，满足毕设答辩最低要求；

- 中优先级：可选模块（性能测试、云端集成、真实传感器对接），均为软件测试系统的扩展能力，可在核心模块完成后迭代，提升测试系统的全面性与含金量（适配考研复试/求职）；

- 低优先级：AI异常检测模块，为软件测试系统的智能升级能力，技术难度高、开发周期长，仅建议在时间充裕时尝试，不作为核心目标。
